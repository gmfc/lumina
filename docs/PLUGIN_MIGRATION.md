# Lumina Plugin Migration — Handoff Spec

> **✅ COMPLETE (historical).** This migration is finished: every user-facing feature listed below
> is now an `editor-builtins` plugin reaching the editor only through `Host`
> (see `crates/builtins/src/lib.rs`). This document is retained as the **design rationale** — the
> Host port catalogue (§4) and per-feature dossiers (§6) record why each port exists — but the
> "remaining / to extract" framing throughout §1 and §6 describes work that has since landed.

_This document was the continuation plan for finishing the plugin-system migration. It complements
[`AUDIT.md`](AUDIT.md) (the pre-migration audit + roadmap) and [`ARCHITECTURE.md`](ARCHITECTURE.md)
(the invariants). The per-feature dossiers below were produced by reading the actual source; they
cite `file:line`._

**Goal:** realize invariants **#3** ("features are plugins") and **#4** ("everything is a
command through one path"). The endpoint is *not* "every command is a plugin" — the core
editing primitives (insert/delete/undo, motions, selection) stay app/core-side as the things
plugins build on. The endpoint is: **every user-facing feature** (find, palette, search, git,
completion, diagnostics, LSP, terminal, vim) is an `editor-builtins` plugin reaching the editor
only through `Host`, and the `Command` table shrinks to those primitives.

## 2. The proven extraction pattern

Follow this for each feature (it's how multi-cursor and git-nav landed):

1. If the feature needs data/effects not on `Host`, add a **focused, additive `Host` method**
   (with a default impl so external/test hosts need not implement it) and override it on
   `EditorState`. Prefer the cohesive port groups in §4 over one-off methods.
2. Create `crates/builtins/src/<feature>.rs`: a struct `impl editor_plugin::Plugin` that **owns**
   the feature's state, contributes its `CommandSpec`s (titles) via `contributions()`, and does
   the work in `run_command` / `on_event` / `render_panel` **through `Host` only**. Add pure
   unit tests.
3. Register it in `editor_builtins::all_builtins_with` (`crates/builtins/src/lib.rs`).
4. Remove the app-side version: the `Command` enum variants (`crates/app/src/input.rs`), the
   `command_for_id` arms (`crates/app/src/commands.rs`), the `dispatch()` arms
   (`crates/app/src/app/dispatch.rs`), the `impl App` methods, and the `palette_entries` rows
   (`crates/app/src/commands/tables.rs`). **Keep** the keybinding rows in `tables.rs`
   (`chord -> id`): the keymap resolves `chord -> id` and `exec_id` routes registry-first, so
   the id reaches the plugin; palette titles now come from the registry.
5. Repoint app tests that drove the feature via `Command::*` to `app.exec_id("<id>")` (the real
   registry path).
6. Generalize `crates/builtins/tests/self_hosting.rs` to guard the new plugin.
7. Keep **every gate green** at each step (see §8).

## 3. Hard constraints

- **Dependency direction** (compiler-enforced): `editor-core` ← `editor-plugin` ← 
  `editor-builtins` ← `lumina`. `editor-plugin` (where `Host` lives) **cannot** reference
  `lumina` types, and `editor-builtins` must not gain `ratatui`/`portable-pty`/`vt100`/
  `lsp-types` deps. All new `Host` types are primitive/`editor-core` types defined in
  `editor-plugin`.
- **The split-borrow wall:** dispatch is `registry.dispatch_command(id, &mut self.editor)`, so a
  plugin's `run_command` only gets `&mut dyn Host` (= `&mut EditorState`). Anything a feature
  needs that lives on `App` (not `EditorState`) must move onto `EditorState` **or** be exposed
  through a new primitive-typed `Host` method. This is why the UI/PTY/LSP ports in §4 are the
  gating work.
- **Keep `Host` focused** (§13 "no god trait"): grow it in the cohesive capability groups of §4,
  not ad-hoc per-command methods.
- **Green at every commit:** one feature per commit, all gates passing (§8).

## 4. The Host port catalogue — build these first

These shared ports unblock the features in §6. Build them as additive, primitive-typed capability groups in `editor-plugin` (types next to `PanelContent`/`Span` in `contribution.rs`), implement them on `EditorState`, and render/consume them generically in `lumina`. Recommended build order: **prompt/overlay input → decorations → picker/command-enumeration** first (they unblock 4+ features), then background-jobs, caret-popup, async-LSP, PTY-panel, key-interception. Also close the keymap gap noted below (the app currently binds only `tables.rs` keybindings, not registry-contributed ones).

_Summary:_ Consolidated Host port catalogue — one coherent, non-god-trait capability-group design in editor-plugin that unblocks migrating find/replace, palette/quick-open, project-search, completion/diagnostics, LSP nav/rename, terminal, and vim into editor-builtins. Eight cohesive capability GROUPS (Prompt, Decorations, Picker+Enumeration, CaretPopup, BackgroundJobs, AsyncLSP, RawPTY, RawKeys) plus the registry-keybinding wiring fix, all primitive/core-typed so builtins need no ratatui/pty/lsp deps.

**Design detail.** Eight capability GROUPS added to Host (host.rs) as additive default-impl methods, each overridden on EditorState (editor.rs impl Host). Grouped, not per-command:

1) PROMPT (grant: ui) — fn set_prompt(&mut self, prompt: Option<PromptSpec>); fn prompt(&self) -> Option<&PromptSpec>. App drives generic text-field editing; plugin gets Event::Prompt{Input,Accept,Cancel} carrying field id+value and runs its submit_command. Replaces overlay.rs:64-124 rename/save-as loops.

2) DECORATIONS (grant: ui) — fn set_decorations(&mut self, doc: DocId, layer: &str, decos: DocDecorations); fn clear_decorations(&mut self, doc: DocId, layer: &str). Layer key = owner (e.g. "find.match","lsp.diag","edit.bracket","git.change"). Plugin recomputes on Event::{DidChange,DidChangeCursor}. Replaces the hardcoded merges in ui/editor.rs:295-342.

3) PICKER + ENUMERATION (grant: ui + commands:run) — fn set_picker(&mut self, picker: Option<PickerSpec>); fn commands(&self) -> Vec<CommandInfo> (mirror of registry command specs pushed onto EditorState at startup so a palette plugin can enumerate through the split-borrow wall). App owns fuzzy filter; delivers Event::Picker{Input(query),Accept(id),Cancel}. Replaces picker.rs + palette.rs.

4) CARET POPUP (grant: ui) — fn set_popup(&mut self, doc: DocId, popup: Option<Popup>) where Popup{ anchor:(usize,usize), rows:Vec<PanelLine>, selected:usize }. Navigation/accept delivered as Event::Popup{Next,Prev,Accept,Cancel}. Replaces completion.rs render/state.

5) BACKGROUND JOBS (grant: jobs — NEW) — fn spawn_job(&mut self, spec: JobSpec) -> JobId; fn cancel_job(&mut self, job: JobId). JobSpec{ kind:String, params:Vec<(String,String)> }; app maps kind ("search","git-diff") to worker.rs spawn_search/spawn_git. Results arrive as Event::Job{Update{job,chunk:JobChunk},Done{job}}. Keeps threads app-side.

6) ASYNC LSP (grant: lsp — NEW) — fn lsp_request(&mut self, req: LspRequest) -> Option<RequestId>. LspRequest{ doc, method:LspMethod, pos:LspPosition, arg:Option<String> }. Reply as Event::LspResponse{ request:u64, result:LspResult } / Event::LspDiagnostics{ doc, diags }. App translates editor-lsp types <-> primitive DTOs. Replaces lsp/requests.rs call sites + handle_lsp_event.

7) RAW PTY (grants: pty — NEW) — fn terminal_open(&mut self, cwd:&Path) -> TerminalId; fn terminal_input(&mut self, id:TerminalId, bytes:Vec<u8>); fn terminal_resize(&mut self, id, rows,cols); fn terminal_close(&mut self, id); fn terminal_grid(&self, id) -> Option<TerminalGrid>. Output as Event::Terminal{Output{id},Exited{id}}. Plugin owns dock/tab lifecycle; app owns PTY + vt100 parse + grid render.

8) RAW KEYS (grant: keys:raw — NEW) — added to the Plugin trait (registry.rs:18): fn capture_key(&mut self, key: Key, host:&mut dyn Host) -> bool; Registry::capture_key offers the key to registered capturers before chord resolution. Key/KeyCode are crossterm-free enums in editor-plugin; app translates. Backs vim + terminal pre-emption (mirrors keys.rs:51 capture_key chain).

**How the app consumes it (rendering / wiring).** The pure renderer stops reading feature-specific fields and instead reads generic, data-described state the ports fill — preserving invariant #2 (render = pure fn of state):
- Prompt: a generic overlay renderer draws PromptSpec.fields (title, labelled text rows, toggle chips, focused field) — one function replacing ui/overlays.rs' per-Overlay arms.
- Decorations: ui/editor.rs cell_style (:295-342) iterates the merged decoration layers for the active doc — Decoration.style is a semantic key resolved through Theme::style_for (same mechanism already used for "git.add"/"bracket.match" at ui/editor.rs:241,325), GutterMark.glyph/style drives draw_git_bar/draw_diag_marker (:226-273) generically. Precedence = layer registration order (or an explicit z field).
- Picker: ui/pickers.rs renders PickerSpec.title + filtered PickerRow list (label + optional detail) + selection; the app-side fuzzy_score (picker.rs:156) becomes the generic filter.
- Popup: a caret-anchored floating list renders Popup.rows (Vec<PanelLine>/Span) at Popup.anchor — same Span styling as panels.
- Terminal: the app renders TerminalGrid (TermCell{ch,style,cursor}) into the panel region; the plugin never produces cells, only lifecycle.
All new DTOs reuse the existing Span/PanelLine vocabulary (host.rs:9-58) so the theme mapping and renderer primitives are shared, not duplicated per feature.

**Data types / relocation.** Per-feature state that moves OUT of EditorState (crates/app/src/editor.rs:44-82) into the owning builtin plugin, with the app keeping only a generic, data-described mirror the renderer reads:
- find: Option<FindState> (editor.rs:65) → FindReplacePlugin owns query/replace/toggles/matches; app keeps only a merged decoration layer + the generic prompt widget.
- picker: Option<Picker> (editor.rs:67) → a PalettePlugin/QuickOpenPlugin owns item sourcing; app keeps a generic PickerSpec + fuzzy filter.
- completion: Option<CompletionState> (editor.rs:74) + nav_locations: Vec<Location> (editor.rs:75) → CompletionPlugin / LspNavPlugin own the lists; app keeps a generic caret Popup.
- diagnostics: HashMap<DocId,Vec<Diagnostic>> (editor.rs:69), bracket_match (editor.rs:72), git_hunks (editor.rs:78) → become decoration layers published via set_decorations; the raw git compute + LSP transport stay app-side.
- Overlay::{RenameInput,SaveAsInput} (editor.rs:32-41) → replaced by a single generic EditorState.prompt: Option<PromptSpec>.
- vim: Option<VimState> (editor.rs:81) → VimPlugin owns mode state; needs raw-key capture (group 8).

New primitive DTO types to ADD in editor-plugin (so builtins need no ratatui/pty/lsp): PromptSpec/PromptField/PromptFieldKind, Decoration/GutterMark/DocDecorations, PickerSpec/PickerRow, Popup (reusing PanelLine/Span), JobSpec/JobId/JobChunk, LspPosition/LspRequest/LspResult + primitive Loc/Completion/Symbol/WorkspaceEdit/Diag mirrors, TerminalId/TerminalGrid/TermCell, and Key/KeyCode (a crossterm-free chord). New EditorState mirror fields the ports write into: prompt, decorations: HashMap<(DocId,String /*layer*/),DocDecorations>, picker (generic), popup, jobs: HashMap<u64,..>, lsp_pending, terminals grid handles.

**Key & event routing.** Today keys route through App::on_key (keys.rs:8-47): capture_key (keys.rs:51-82) gives modal overlays (handle_modal_key keys.rs:86-99 → overlay_key/picker_key/search_key/find_key), terminal focus, completion popup, sidebar, settings, and vim first refusal; only unclaimed keys hit chord resolution (self.keymap.resolve) then text_entry_fallback. Every one of those branches reads a bespoke App/EditorState field.

Plugin routing needed:
- Modal input (prompt/picker/popup): while EditorState.prompt/picker/popup is Some, the app runs the GENERIC editing (type/backspace/toggle-field/move-selection/accept/cancel) itself and emits Event::{Prompt,Picker,Popup}* to the owning plugin — the plugin never sees raw keys for these; it only reacts to semantic events. This collapses overlay_key/picker_key/find_key/completion_key into one generic modal handler.
- Pre-emptive raw capture (vim, terminal): needs the new Plugin::capture_key(Key,&mut Host)->bool offered before keymap.resolve (a registry hook slotted into capture_key at keys.rs:51). Returns true to consume. Key must be a primitive editor-plugin type; App translates crossterm KeyEvent -> Key. Terminal input bytes still go through Host::terminal_input, computed app-side (terminal::key_to_bytes stays app-side, keys.rs:141).

**Capabilities / commands.** Registry keybinding wiring (the identified gap): build_keymap (app.rs:140-146) currently layers only default_bindings() (tables.rs:83) + config.keybindings and IGNORES registry.keybindings() (registry.rs:135, populated from Contributions.keybindings contribution.rs:66-79). Fix: change build_keymap to take &Registry and fold three tiers in precedence order defaults < registry-contributed < user config:
  let mut km = Keymap::from_pairs(default_bindings());
  for kb in registry.keybindings() { km.bind(&kb.chord, &kb.command); }
  for (chord,id) in &config.keybindings { km.bind(chord,id); }
Call sites app.rs:141, lifecycle.rs:62 (construction; registry is already in scope, App{registry} built same fn), and reload_config lifecycle.rs:123 must pass &self.registry. Keymap::bind already gives last-writer-wins (keymap.rs:172), so config still overrides.

Once wired, each migrated plugin contributes its own chords via Contributions::builder().keybinding(...) and its default rows are REMOVED from tables.rs:default_bindings() (currently git.nextHunk alt+j/alt+k at tables.rs:177-178, cursor.addNextMatch ctrl+d :116, cursor.addAbove/Below :169-170, selectAllMatches :171 still live in tables.rs even though the features are already plugins — move them to the plugins' contributions()). Palette titles already come from the registry (palette.rs:50-55), so palette_entries() rows for a migrated feature are deleted (as already done for multicursor/git-nav, tables.rs:37-39,61). No new Command enum variant or command_for_id arm is added for a plugin; exec_id (overlay.rs:13-21) already routes registry-first. For features still partly app-backed (find/search/lsp/terminal), keep their Command arms until the plugin fully owns the id.

**What stays app/core-side.** - The actual transports: PTY spawn + vt100 parsing + byte-budget draining (worker.rs:26-29, workers.rs:61-69, terminal/*), the LSP client/process management + JSON-RPC correlation (lsp.rs:47-140, lsp/requests.rs), git shell-out (git.rs, worker.rs:110-115), and the notify file-watcher (worker.rs:66-98). Plugins express intent through primitive ports; the app executes I/O and threading. This is the split-borrow constraint: run_command only gets &mut dyn Host, and none of these can be primitive-typed on that surface.
- The generic renderers: the ratatui overlay/picker/popup/decoration/terminal-grid draw code (ui/*.rs) stays app-side and consumes the DTOs; builtins produce data, never cells (keeps ratatui out of editor-builtins).
- Core edit primitives: Transaction application, Selections normalization, undo history (editor.rs:179-202) — edits still flow through Host::apply_transaction/set_selections (invariant #1/#2 already hardened).
- The keymap engine (keymap.rs) and the crossterm KeyEvent->Key/Chord translation stay app-side; the registry only supplies chord->id bindings and capture hooks.
- The event fan-out + pending-queue draining (workers.rs:8-42) and capability gating for external guests (runtime/dispatch.rs:60-81, wasm.rs) stay app/kernel-side; new ports just add gated action kinds.
Rationale: each of these is either non-primitive (ratatui/pty/lsp/git types) or a genuine app policy (file IO, threading, sandbox), so per §13 they must not leak into the Host trait; the ports expose only primitive data + intent.

## 5. What stays app/core-side (and what still migrates)

The dividing line: **editing primitives and genuinely app-level IO stay**; **features migrate.**

_Summary:_ App-level commands + editing primitives: what STAYS vs what MIGRATES to plugins. PRINCIPLE / roadmap endpoint: the Command enum + dispatch() shrink to the editing primitives that mutate the active Document (motions, selection-extend, insert/delete/line-ops/undo/redo). These ARE the API plugins invoke via Host::apply_transaction / set_selections, so they legitimately stay app/core-side; invariant #4 (one path) is already satisfied because every id resolves through exec_id (registry-first, then the command table) -- a primitive need not be a plugin to be a command. What migrates are features layered on primitives but hardcoded today: theme toggle and vim (audit finding #1 names both), one stray selection-only command (CursorsToLineEnds) that belongs in the existing multicursor plugin, and clipboard once a register capability + an edit-over-Host helper exist. Files / tabs / settings / config.reload STAY: filesystem I/O and App-owned-resource lifecycle, not editor-state transforms.

### Proposed ports / methods

| Port group / method | Proposed signature | Purpose |
|---|---|---|
