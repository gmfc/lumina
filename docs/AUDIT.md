# Lumina Architecture Audit

_A systematic audit of the Lumina codebase against [`docs/ARCHITECTURE.md`](ARCHITECTURE.md) —
its nine load-bearing invariants and thirteen guideline sections — with a focus on the plugin
system (invariants #3/#4) and overall modularity. Every finding below was produced by an
independent reader, then adversarially re-verified against the source before inclusion._

**Method.** Ten dimensions were audited in parallel; each finding was then checked by a second,
skeptical pass instructed to *refute* it against the code. 30 findings survived verification;
1 was refuted and dropped. Two dimensions (dependency direction §2, concurrency §9) were audited
directly and found healthy.

---

## Executive summary

Lumina's low-level foundations are genuinely strong, but its headline architectural promise — invariant #3/#4, "features are plugins" and "everything is a command" — is aspirational, not implemented. The core is the healthiest part of the tree: editor-core is provably headless (zero crossterm/ratatui/io deps; only ropey/unicode/slotmap), the Transaction engine is correct and reversible, coordinate mapping is pure, §9's one-Mutex/single-owner rule holds, and §12 (unsafe forbidden, clippy denied, CI gates present) is clean. The external plugin substrate is also mostly real: Rhai and WASM both implement the same Plugin trait, load into the same Registry, route through registry.dispatch_command, and WASM guests genuinely run deny-by-default (empty Linker, fuel-metered). So the machinery for a plugin-centric editor exists and works — for exactly one plugin.

The gap is that the machinery is unused by the app itself. There are effectively THREE command systems, not one: the Registry (the intended single path, carrying only the explorer's 6 commands), a hardcoded id→Command enum funneled through a 165-line dispatch() mega-match that mutates EditorState directly (~99 ids), and a stringly-typed match in exec_id for 6 more app-level ids (vim, theme, settings) that touch neither the enum nor the Registry. So ~105 built-in command ids bypass the Registry while 6 go through it, and the precedence is hardcoded-first/registry-last — the inverse of #4. editor-builtins registers only ExplorerPlugin; the find/palette/search/lsp plugins its own comment promises "phase by phase" were never built, and the self-hosting test that supposedly guards #3 asserts only about the explorer with a non-regression loop that never executes. The root cause is structural, not lazy: the Host port is a clean 9-method surface built to exactly the explorer's needs and stopped there. It cannot express find, pickers, LSP requests, decorations, terminal, git, or vim, and half the feature state lives on App (not the Host object EditorState), so those features cannot be plugins even in principle. Widening Host is therefore the true unblocker; everything else is sequencing behind it. Separately, two correctness bugs unrelated to plugin-ization deserve attention regardless: lossy decode silently corrupts non-UTF-8 files on save (invariant #6 data loss), and non-transactional set_text on external reload leaves undo history able to replay stale-offset transactions.

## Top findings (verified, prioritized)

### 1. 🔴 CRITICAL — Three parallel command systems: ~105 built-in ids bypass the Registry, only 6 go through it
*Invariant/section: #4 EVERYTHING IS A COMMAND via one path; #3 built-ins register through the SAME contribution API, no privileged back doors*

**Evidence.** crates/app/src/commands.rs:15-129 maps 99 ids to a Command enum; crates/app/src/app/dispatch.rs:13-177 is a monolithic match mutating self.editor directly. Only dispatch.rs:172 (Command::Run) routes to registry.dispatch_command, reached only for plugin ids. exec_id (crates/app/src/app/overlay.rs:9-28) adds a third path: command_for_id first, then a raw string match for config.reload/view.toggleTheme/vim.*/view.settings, registry last. crates/builtins/src/lib.rs:23 registers ONLY ExplorerPlugin (6 commands).

**Why it matters.** The plugin system Lumina is built around is used by exactly one feature. Precedence is hardcoded-first/registry-last, so a plugin can never override or intercept a built-in id, and the 'same path for built-in and third-party' that #3 promises does not exist. This is the central architectural drift the whole tree is organized to prevent.

**Remediation.** Collapse to one path: make built-ins contribute commands through Contributions like the explorer, invert exec_id/dispatch to resolve every id through registry.dispatch_command first, and shrink the Command enum + dispatch() to the core editing primitives plugins invoke via Host. Fold the 6 stringly-matched app-level ids into registry contributions (vim and theme should become plugins).

### 2. 🟠 HIGH — The Host port is too thin to express any app feature — plugin-ization is structurally blocked
*Invariant/section: §7 Host is the imperative surface a plugin drives; #3 requires that surface be wide enough for built-ins to reach the editor only through it*

**Evidence.** Host exposes 9 methods (crates/plugin/src/host.rs:77-114): workspace/active_doc/root/apply_transaction/set_selections/open_path/read_dir/set_panel/set_status/notify/execute. The hardcoded features call app internals unreachable through Host: lsp_request (dispatch.rs:134-142), open_find/replace_all (dispatch.rs:106-117), open_palette/open_quick_open (dispatch.rs:151-152), toggle_terminal (dispatch.rs:156-159), goto_hunk (dispatch.rs:144-145). No Host method exists for find, LSP, pickers, overlays, decorations, terminal, git, or background jobs.

**Why it matters.** This is the structural root cause behind the command-system drift and the never-delivered builtins: even in principle these features cannot be plugins because the port cannot express them. The 'no god trait' checkmark (§13) masks that Host is focused because it is under-powered, not because features are cleanly declarative. Every extraction is blocked until Host grows.

**Remediation.** Grow Host in focused, single-purpose capability groups (avoid a real god trait): an overlay/prompt-input port, an open_picker(items)+commands() enumeration port, an add_decorations(doc,kind,spans) port for gutter/inline, a spawn_job/message port, and an async LSP request/response port. Each port addition unblocks exactly one feature extraction.

### 3. 🟠 HIGH — editor-builtins never delivered the promised plugins; the self-hosting guardrail is vacuous
*Invariant/section: #3 features are implemented AS plugins over the contribution API; the self-hosting test proves disabling one removes exactly its contributions*

**Evidence.** crates/builtins/src/lib.rs:20-24 returns vec![ExplorerPlugin] with a comment listing find/replace/palette/quick-open/search/sync/lsp as to-be-added — they never were. crates/builtins/tests/self_hosting.rs hardcodes EXPLORER_* ids; its non-regression loop filters before.iter() for !id.starts_with("explorer."), but `before` contains only explorer ids, so the loop body never executes. The file is also gated behind #![cfg(feature = "proptests")].

**Why it matters.** The one test meant to prove the load-bearing invariant proves nothing beyond the explorer and its key assertion is a no-op — it cannot fail if find/lsp/palette are wired directly into app, because none of them are registered as plugins for it to observe. The invariant is unguarded.

**Remediation.** As features migrate to plugins, generalize the test to assert every user-facing palette/keymap id resolves through the Registry (command_for_id returns None for them) and make the disable-isolation check operate on a multi-plugin set so the loop is non-empty. Until then, amend lib.rs and ARCHITECTURE.md to state honestly that only the explorer is self-hosted today.

### 4. 🟠 HIGH — Lossy decode silently corrupts non-UTF-8 files on save (data loss)
*Invariant/section: #6 file encoding preserved verbatim on save*

**Evidence.** crates/app/src/files.rs:51-64 decode uses String::from_utf8_lossy; files.rs:66-78 decode_utf16 uses from_utf16_lossy and chunks_exact(2) (dropping a trailing odd byte). Saved bytes come from encode/serialize (files.rs:94-115) which re-encode the already-lossy String. There is no original-bytes buffer.

**Why it matters.** Any byte not valid in the detected encoding (latin-1 text, partially-binary files) is replaced with U+FFFD at load and re-emitted on save, irreversibly corrupting the on-disk file with no warning. This is silent data loss on a plain save — independent of the plugin goals but a direct violation of a load-bearing invariant.

**Remediation.** Keep the raw original bytes and write unchanged regions back byte-for-byte, or detect undecodable input and mark the document read-only / require explicit acknowledgment before saving. At minimum surface a non-silent warning before writing a lossily-decoded buffer. Never round-trip U+FFFD to disk silently.

### 5. 🟠 HIGH — Document.text is a public Rope field — invariant #1 has no compile-time guard
*Invariant/section: #1 all buffer mutation goes through a reversible Transaction; raw rope helpers private to core; §13 raw rope mutation is a forbidden anti-pattern*

**Evidence.** crates/core/src/document.rs:22 declares pub text: Rope. Read access is used across app/plugin/builtins (find.rs:88, runtime/dispatch.rs:29, etc.), proving it is reachable outside core; nothing stops doc.text.insert(..)/doc.text = .. from those sites. The mutating apply_raw_* helpers are correctly pub(crate) (mutate.rs:37,60), but the public field defeats them.

**Why it matters.** The #1 guarantee is upheld only by developer discipline. Any workspace crate could mutate the rope directly, skipping Transaction, history, the dirty flag, the revision bump, and the tree-sitter edit stream — silently corrupting undo and incremental highlighting. The compiler would accept it. pub selections similarly lets outside code install a non-normalized set (#2).

**Remediation.** Make text pub(crate) and add pub fn rope(&self)->&Rope plus the existing char/line/slice helpers so the ~30 read sites keep compiling. Audit the other bare-pub Document fields (selections, history) the same way. This converts #1 from a review-time convention into a compile-time guarantee — a pure refactor that keeps the build green.

### 6. 🟠 HIGH — set_text/set_text_str is a pub non-transactional whole-buffer mutation; reload leaves undo stale
*Invariant/section: #1 transactions are the only way a buffer changes; mutation must be reversible*

**Evidence.** crates/core/src/document/mutate.rs:86,94 assign self.text directly, bumping revision but recording no Transaction. Called from crates/app/src/app/workers.rs:158 on external reload, which never clears doc.history (History::undo/redo at history.rs:119/129 are not invoked).

**Why it matters.** A pub path replaces the whole rope with no inverse and no undo entry — directly contradicting 'the only way'. After reload, a later undo replays transactions whose offsets were computed against the pre-reload buffer; apply_raw_* clamps so it won't panic, but undo restores wrong text — a correctness bug. The pub visibility is also an escape hatch any plugin/builtin can call.

**Remediation.** Narrow set_text/set_text_str to pub(crate) (or move reload into core behind a named seam), and on whole-buffer replace either model it as a Transaction or explicitly clear/rebase doc.history so post-reload undo cannot replay stale-offset transactions.

### 7. 🟠 HIGH — Feature state is split across two god-objects; the App half is unreachable through Host
*Invariant/section: #3 features reach the editor only through editor_plugin::Host; §13 state has a single owner, no god object*

**Evidence.** editor.rs:44-82 EditorState carries 19 fields (find, picker, diagnostics, completion, git_hunks, vim, ...); app.rs:42-97 App carries ~25 more including the terminal panel, LSP manager, project search, and settings. Host is implemented on EditorState only (editor.rs:174), so terminal/lsp/search/settings are invisible to plugins. self.panel is touched across 7 files.

**Why it matters.** State a plugin would need to own is scattered between the Host object and App. Terminal, LSP, project-search, and settings have no Host accessor, so even a native builtin plugin has no API to reach them — they cannot be plugins even in principle without new ports plus a state relocation. EditorState is itself a 19-field grab-bag mixing view state, syntax caches, and per-feature widgets.

**Remediation.** Give each feature a single owning struct reached through the Registry, not App fields. Model live-state features (terminal, lsp) as plugin-owned state driven by Host ports (jobs, requests, decorations). Shrink EditorState to core view/workspace state; move per-feature widgets (find, picker, completion) into their plugins.

### 8. 🟡 MEDIUM — read_dir — the doc's named capability seam — gates nothing and is unreachable by external plugins
*Invariant/section: §7 read_dir is the capability seam; external plugins are deny-by-default and the host enforces fs:read grants*

**Evidence.** ARCHITECTURE.md:200,212-213 name read_dir as the fs:read seam. But Host::read_dir (host.rs:101) takes only (&self, path) — no caller identity or capability — and the app impl (editor.rs:206-222) calls std::fs::read_dir unconditionally. No external action maps to read_dir: the dispatch tables handle only insert/replace/notify/open/run/set_panel (runtime/dispatch.rs:55-82, wasm.rs:231-257). fs:read actually gates the `open` action → host.open_path, which the guide never mentions.

**Why it matters.** The guide's central fs:read enforcement claim is false in code. The real capability seam is the action-marshaling match in each runtime (has_cap gating), not read_dir. A reader trusting §7 would misunderstand where the security boundary actually is.

**Remediation.** Either route external read_dir through a has_cap("fs:read")-gated action mirroring the `open` arm (giving the host the caller's cap set), or correct ARCHITECTURE.md/host.rs:101 to state that enforcement lives at the guest action boundary and read_dir is an internal native-only host helper. Also add commands:run and the open/run/set_panel actions to the §7 capability/action list to match the code.

### 9. 🟡 MEDIUM — render() is not pure: top-level draw() takes &mut App and mutates state during layout
*Invariant/section: #8 rendering is a pure function of state — render(&State)->frame mutates nothing*

**Evidence.** crates/app/src/ui.rs:39 pub fn draw(f: &mut Frame, app: &mut App); line 59 writes app.page_height, lines 95-102 write app.regions. The module doc (ui.rs:1-2) asserts the opposite. All sub-renderers correctly take &App, so the violation is localized to the entry point.

**Why it matters.** The single render entry point cannot satisfy render(&State), contradicting a load-bearing invariant and the file's own doc. Impact is behaviorally benign (deterministic geometry caching) but the signature is a genuine break in the pure-render contract.

**Remediation.** Make draw take &App and return a small FrameLayout { page_height, regions }; have the event loop store those into App after draw returns. Restores the pure render(&State)->frame shape and makes the doc comment true. Small, low-risk change.

### 10. 🟡 MEDIUM — Selection normalization is convention, not parse-don't-validate; one edit path lacks the overlap clamp
*Invariant/section: #2 a document holds a normalized, sorted, non-overlapping SET; normalize at the boundary*

**Evidence.** selection.rs:5 doc says normalize 'call it religiously'. push/single/set_single don't normalize; only from_iter/normalized/transform do. Boundary setters assign verbatim: editor.rs:197 (Host::set_selections) and document.rs:139 (Document::set_selections). edit/apply.rs:14 edit_selections calls clamp_non_overlapping, but edit/apply.rs:89 edit_selections_sel only sorts and has NO overlap clamp while relying on the set being non-overlapping.

**Why it matters.** The seam a plugin drives (Host::set_selections) accepts an arbitrary Selections, so a caller that built one via single+push injects an unsorted/overlapping set; if it then drives the auto-pairs path, overlapping selections yield overlapping Transaction changes — the exact buffer+undo corruption apply.rs warns about. In-tree callers normalize by discipline, so no live bug — latent gap plus drift.

**Remediation.** Normalize at the boundary: have Host::set_selections and Document::set_selections call .normalized() (or take a NormalizedSelections newtype so an unnormalized set is unrepresentable), and add clamp_non_overlapping to edit_selections_sel so it matches its sibling.

## Modularization roadmap

An ordered sequence of **independently shippable** steps toward *"everything uses the plugin
system"* and *"the system is itself modular."* Each step keeps the build green. Steps 1–2 need
no `Host` changes; from Step 3 on, each feature extraction is unblocked by one focused, additive
`Host` capability-group port (so the port never becomes a god trait).

### Step 1 — Lock the mutation and selection seams into the type system so later plugin extractions can be trusted (safest, highest-leverage foundation)
`risk: low` · `effort: M` · **Host additions:** none

Make Document.text pub(crate) and add pub fn rope(&self)->&Rope plus char/line/slice accessors; migrate the ~30 external read sites. Narrow set_text/set_text_str to pub(crate) and clear/rebase doc.history on external reload (workers.rs:158). Normalize inside Host::set_selections and Document::set_selections and add clamp_non_overlapping to edit_selections_sel. Pure refactors; behavior unchanged; build stays green.

### Step 2 — Fix doc-vs-code drift and the two correctness bugs that are independent of plugin work, so the guide stops overstating and files stop corrupting
`risk: low` · `effort: M` · **Host additions:** none

Amend editor-builtins/src/lib.rs and ARCHITECTURE.md to state only the explorer is self-hosted today; correct §7/host.rs:101 to say enforcement lives at the guest action boundary and list commands:run + open/run/set_panel. Fix lossy decode (files.rs) to preserve original bytes / refuse silent U+FFFD save. Make draw() take &App and return FrameLayout. Drop the proptests gate (or add a deterministic reversibility test) and add a clippy --features proptests invocation; refresh stale ci.yml scaffold comments.

### Step 3 — Widen Host with the first, additive capability-group ports needed by leaf features — no behavior change, unblocks extractions
`risk: low` · `effort: L` · **Host additions:** overlay/prompt-input; open_picker + commands() enumeration; add_decorations

Add single-purpose ports to the Host trait and implement them on EditorState/App: an overlay/prompt-input port, open_picker(items, on_select) + commands() enumeration, and add_decorations(doc, kind, spans) for match-highlight/gutter. Nothing consumes them yet; the app keeps its current hardcoded paths. Purely additive to the trait.

### Step 4 — Extract FIND/REPLACE as the first real second plugin, proving the pattern end-to-end
`risk: medium` · `effort: L` · **Host additions:** none (uses step 3 ports)

Move find.rs logic into an editor-builtins FindPlugin that drives set_selections (already in Host) + the overlay and decoration ports from step 3; contribute its command ids via CommandSpec. Register it in all_builtins. Remove the find/replace Command variants and dispatch arms. Generalize the self-hosting test to assert find ids resolve through the Registry and that disabling FindPlugin removes exactly its contributions.

### Step 5 — Extract PALETTE / QUICK-OPEN / GOTO-LINE, then invert dispatch precedence to registry-first
`risk: medium` · `effort: L` · **Host additions:** none (uses step 3 ports)

Move palette.rs into a plugin using commands() + open_picker + open_path/execute (already available). Once two-plus real plugins carry commands, invert exec_id/dispatch: resolve every id through registry.dispatch_command first, falling back to app-native handlers only for genuine app-level concerns. Fold the 6 stringly-matched ids (config.reload, view.toggleTheme, vim.*, view.settings) into registry contributions — theme becomes a small plugin.

### Step 6 — Extract GIT GUTTER and DIAGNOSTICS/COMPLETION, adding the background-job and caret-popup ports
`risk: medium` · `effort: L` · **Host additions:** spawn_job/message port; caret-anchored popup port

Add a spawn_job/message port and a caret-anchored popup port to Host. Move git.rs navigation (pure over git_hunks) and the completion/diagnostics popups into plugins driving the job + decoration + popup ports. Relocate git_hunks/diagnostics/completion state off EditorState into plugin-owned state. Retire their Command variants; extend the self-hosting test.

### Step 7 — Extract PROJECT SEARCH onto the job + results-panel ports
`risk: medium` · `effort: L` · **Host additions:** results-panel input (reuses set_panel + spawn_job)

Move search.rs and last_search_run off App into a plugin that uses the spawn_job port (step 6) and set_panel for results. Remove its dispatch arm. This is the first App-owned (non-EditorState) feature to migrate, validating that plugin-owned live state works.

### Step 8 — Extract LSP behind an async request/response Host port
`risk: high` · `effort: XL` · **Host additions:** async LSP request/response port; diagnostics ingestion

Add an async LSP request/response + diagnostics-ingestion port to Host. Move LspManager and lsp_sent_revision off App into a plugin driving that port; feed diagnostics through the decoration/popup ports from step 6. Retire Hover/GotoDefinition/Completion dispatch arms. Largest port addition so far; do after the decoration/job infrastructure exists.

### Step 9 — Extract TERMINAL onto a raw panel + input port
`risk: high` · `effort: L` · **Host additions:** raw panel + keystroke-input port (PTY-capable)

Add a raw panel + keystroke-input port to Host. Move TerminalPanel off App into a plugin owning PTY lifecycle behind that port. Retire ToggleTerminal/NewTerminal dispatch arms.

### Step 10 — Extract VIM last behind a raw key-interception port
`risk: high` · `effort: XL` · **Host additions:** key-filter (raw key interception) contribution; selection/motion query access

Add a key-filter contribution port (intercept raw key events before chord resolution) plus the selection/motion access vim needs. Move the 1817-line vim.rs impl App layer and VimState into a plugin. Until this port exists, keep vim in-app but document it in the builtins roadmap as a known non-plugin so the self-hosting claim isn't silently overstated. After migration, the Command enum + dispatch() should be reduced to core editing primitives only.

## Full findings catalogue

All 30 verified findings, grouped by audit dimension, most-severe first within each group.

### #3, #4

_Invariants #3 (features are plugins) and #4 (everything is a command) are the most badly drifted dimension in the tree. There are effectively THREE command systems, not one: (a) the Registry — the intended single path, but it carries only the explorer's 6 commands; (b) a hardcoded id→Command table (crates/app/src/commands.rs, 99 arms) funneled through a monolithic 180-line dispatch() match that mutates EditorState directly; and (c) six app-level ids (config.reload, view.toggleTheme, vim.*, view.settings) resolved by a stringly-typed match in exec_id that never touch either the Command enum OR the registry. So ~105 built-in command ids bypass the Registry entirely while 6 go through it. The catalog is lopsided: every headline feature — editing, find/replace, palette, quick-open, project search, LSP, git hunks, terminal, tabs, vim — is a hardcoded dispatch() path; only the file explorer is a genuine Registry-contributed plugin. The self-hosting test that is supposed to guard #3 proves nothing beyond the explorer and its key assertion is vacuous. ### #1

_Invariants #1 and #2 are upheld in-tree by discipline, and the transaction engine itself is correct and well-tested (reversible, bottom-up apply, canonical inverse ordering — transaction.rs). The `apply_raw_*` chokepoint is correctly `pub(crate)` and only reached from `Transaction::apply`. But the enforcement is convention, not the type system: the rope is a public field and there is a `pub` whole-buffer replacement that bypasses transactions entirely, so #1's "raw rope mutation impossible outside core" is not actually guaranteed. #2's normalization is likewise a "call it religiously" convention rather than parse-don't-validate — the boundary setters don't normalize, and one edit path doesn't defend against overlap. No live raw-rope mutation leak exists in app/builtins/plugin today (all reads), but the seams that would let one in are open._

- **🟠 HIGH — Document.text is a public Rope field — the invariant #1 chokepoint is bypassable by any crate** (#1)
  - _Guide:_ ARCHITECTURE.md #1 and §13: all buffer mutation goes through a reversible Transaction; raw rope helpers are private to core; raw rope mutation is a named anti-pattern. document.rs:4 itself says 'never poke the rope directly from outside core'.

Any code outside core can mutate the rope directly, skipping Transaction, history recording, the `dirty` flag, the `revision` bump, and the tree-sitter SyntaxEdit stream — silently corrupting undo and incremental highlighting.
  - _Evidence:_ crates/core/src/document.rs:22 `pub text: Rope`. Reads flow through it everywhere outside core: crates/app/src/app/vim.rs:659/743/857/1023/1092, crates/app/src/app/cursors.rs:22/94/145, crates/app/src/app/completion.rs:20/27/83/110, crates/app/src/app/find.rs:88/104/141/185, crates/app/src/app/lsp.rs:180, crates/plugin/src/runtime/dispatch.rs:29.   - _Fix:_ Make `text` private; expose a read-only accessor (`fn rope(&self) -> &Rope`, plus the existing slice/char/line helpers) so the ~30 read sites keep working, and force every mutation through Transaction. This turns #1 into a compile-time guarantee instead of a review-time convention.
- **🟠 HIGH — set_text / set_text_str is a pub non-transactional whole-buffer mutation; external reload leaves undo history stale** (#1)
  - _Guide:_ ARCHITECTURE.md #1: transactions are 'the only way a buffer changes'; mutation must be reversible; raw helpers private to core. transaction.rs:1 restates 'the only way a buffer changes'.
  - _Code:_ Provides a `pub` (crate-external) path that replaces the whole rope with no inverse and no undo entry, directly contradicting 'the only way'. Because history is not rebased/cleared on reload, a later undo replays transactions whose `at`/`removed` offsets were computed against the pre-reload buffer. apply_raw_* clamps offsets so it won't panic, but the undo restores wrong text — a correctness bug. The `pub` visibility also lets any plugin/builtin call it, an escape hatch around invariant #1.
  - _Evidence:_ crates/core/src/document/mutate.rs:86 `pub fn set_text(&mut self, rope: Rope)` and :94 `pub fn set_text_str(&mut self, s: &str)` assign `self.text = rope` directly (line 87), bumping revision but recording no Transaction.   - _Fix:_ Narrow `set_text`/`set_text_str` to `pub(crate)` (or move the reload into core behind a named seam), and on whole-buffer replace either model it as a Transaction or explicitly clear/rebase `doc.history` so post-reload undo cannot replay stale-offset transactions.

### #8

_Coordinate mapping (#6) is genuinely pure and healthy: screen_to_char/char_to_screen and the column functions in crates/core/src/view.rs take everything by reference, return values, and do no caching mutation; the syntax cache read during render (line_spans(&self)) is also read-only. Atomic save (#9) is correctly implemented as temp-write + sync + rename in the same directory. The two real problems are: (a) invariant #8 is violated at the top level — draw() takes &mut App and writes back app.page_height and app.regions during rendering, directly contradicting both the invariant and the file's own "no mutation" claim; and (b) invariant #6's "encoding preserved verbatim" is broken because decode is lossy (invalid UTF-8/UTF-16 bytes are replaced with U+FFFD and then written back on save, silently corrupting the file). Line-ending faithfulness is also only approximate (mixed/CR-only endings get normalized)._

- **🟠 HIGH — render() is not pure: top-level draw() takes &mut App and mutates state during rendering** (#8)
  - _Guide:_ Invariant #8: rendering is a pure function of state — render(&State)->frame mutates nothing.
  - _Code:_ The render entry point receives &mut App and, as a side effect of laying out the frame, writes two pieces of state back into App (page_height, consumed next tick by PageUp/PageDown; regions, consumed by the mouse router outside draw). All sub-renderers (render_editor/sidebar/status/etc.) correctly take &App, so the violation is localized to the single entry point, but it is a genuine contradiction of a load-bearing invariant and of the file's own doc comment. Impact is behaviorally benign (deterministic geometry caching, no double-render hazard) but the signature cannot satisfy render(&State).
  - _Evidence:_ crates/app/src/ui.rs:39 `pub fn draw(f: &mut Frame, app: &mut App)`; line 59 writes `app.page_height = main_body.height...`; lines 95-102 write `app.regions = Regions {...}`. The module doc at ui.rs:1-2 asserts the opposite: "Rendering — a pure function of state (plan §4, invariant #8). No mutation of editor state happens here; we only read it and write cells."
  - _Fix:_ Make draw take &App and return a small value type carrying the layout outputs (e.g. `struct FrameLayout { page_height: usize, regions: Regions }`); have the caller (the event loop) store those into App after draw returns. This restores the pure render(&State)->frame shape and makes the doc comment true.
- **🟠 HIGH — Encoding not preserved verbatim: lossy decode corrupts invalid-UTF-8/odd-length files on save** (#6)
  - _Guide:_ Invariant #6: file encoding is preserved verbatim on save.
  - _Code:_ Any byte sequence that is not valid in the detected encoding is replaced at load time with U+FFFD (and, for odd-length UTF-16, a trailing byte is silently dropped). Because there is no separate original-bytes buffer, a subsequent save re-emits the replacement characters, silently and irreversibly corrupting the on-disk file. This turns a save of any file containing invalid UTF-8 (latin-1 text, partially-binary files) into data loss with no warning. The load-time comment at files.rs:50 ("Falls back to lossy UTF-8 so we never fail to open a file") acknowledges the lossy open but nothing guards the save.
  - _Evidence:_ crates/app/src/files.rs:51-64 `decode` uses `String::from_utf8_lossy` for the UTF-8 and UTF-8-BOM paths; crates/app/src/files.rs:66-78 `decode_utf16` uses `String::from_utf16_lossy` and `chunks_exact(2)` (dropping any trailing odd byte). The saved bytes come from `encode`/`serialize` (files.rs:103-115, 94-100) which re-encode this already-lossy String.
  - _Fix:_ Either (a) detect undecodable bytes and mark the document read-only / refuse to save through the normal path until the user acknowledges, or (b) keep the raw original bytes and, when the buffer is unmodified in a region, write them back byte-for-byte; at minimum surface a non-silent warning before writing a lossily-decoded buffer. Do not silently round-trip U+FFFD to disk.
- **🟡 MEDIUM — Line endings normalized rather than preserved verbatim for mixed / CR-only files** (#6)
  - _Guide:_ Invariant #6 / types.rs:14-15 own comment: line terminator style is "Preserved from the original file; never silently rewritten."
  - _Code:_ A file with mixed LF/CRLF endings is collapsed to one uniform style on save (every '\n' either kept or turned into '\r\n'), so unchanged lines do not round-trip byte-for-byte. Classic Mac CR-only endings are not modeled at all — lone '\r' survives in the rope but is never recognized as a line ending, so line counting and save behavior are undefined for such files. This is silent rewriting, which the type's own doc explicitly promises not to do.
  - _Evidence:_ crates/core/src/document/types.rs:16-42: `LineEnding` models only `Lf` and `Crlf`; `detect` picks a single dominant style. crates/core/src/document.rs:55-57: `from_str` normalizes with `s.replace("\r\n", "\n")` (lone `\r` left as-is, no CR-only model). crates/app/src/files.rs:94-100 `serialize` re-emits every newline as the one detected style.
  - _Fix:_ At minimum document that only pure-LF and pure-CRLF are round-trip faithful and that mixed files are normalized; better, add a `Cr`/`Mixed` variant (or preserve per-line terminators) so files that were not uniformly LF/CRLF are not silently rewritten on save.
- **⚪ LOW — Atomic save lacks parent-directory fsync and leaves temp file on rename failure** (#9)
  - _Guide:_ Invariant #9: saves are atomic (temp-write + rename).
  - _Code:_ Atomicity (no reader ever sees a partial file) is correctly achieved. But durability across a crash is not guaranteed: without fsync of the containing directory the rename may not be persisted, so a crash right after save can leave the old file or no visible rename despite the data having been synced. On a failed rename the `.lumina.tmp` file is orphaned. Writing through a symlinked path replaces the symlink with a regular file rather than following it. These are edge-case robustness gaps, not violations of the stated atomic-rename invariant.
  - _Evidence:_ crates/app/src/files.rs:132-151 `save`: creates `<name>.lumina.tmp` in the same dir, `f.sync_all()` (line 142) syncs the file data, then `fs::rename(&tmp, path)` (line 148). No fsync of `dir` after the rename; on rename error the temp file is not removed; if `path` is a symlink the rename replaces the link itself.
  - _Fix:_ After the rename, open the parent directory and fsync it to make the rename durable; on any error path remove the temp file (or use a scope guard); if symlink-following semantics are desired, canonicalize `path` before choosing the temp location and rename target.

### §7

_The external plugin substrate is mostly healthy on its headline claims: both Rhai (ScriptPlugin, runtime.rs) and WASM (WasmPlugin, wasm.rs) implement the same Plugin trait, load via runtime::load_dir into the same Registry, and route commands through registry.dispatch_command — no substrate bypasses the registry, and the app actually wires them in (lifecycle.rs:18-24). ### #3 (features are plugins) / #4 (everything is a command)

_The app crate's INTERNAL modularity is a study in contrasts. Cohesion is actually decent: features are split into ~19 focused `impl App` submodules (app/find.rs, git.rs, lsp.rs, completion.rs, terminal, vim.rs, etc.), each backed by a self-contained state struct (FindState, Picker, LspManager, TerminalPanel, VimState, SettingsView) — these are clean, extractable seams, not spaghetti. But the crate fails the modularity test that actually matters for this project (invariant #3, "features are plugins"): every one of these features is wired as a hardcoded `Command` enum variant → a single ~165-line `dispatch()` mega-match → direct mutation of `App`/`EditorState`, NOT through the Registry/Host contribution API. Adding or removing a feature touches 4+ files and cannot be done by disabling a plugin. - **🟠 HIGH — Every built-in feature is a hardcoded Command variant + dispatch() mega-match, not a plugin — invariant #3 never reached** (#3 (features are plugins) / #4 (everything is a command))
